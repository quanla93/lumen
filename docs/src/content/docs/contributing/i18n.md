This page is the **contributor** counterpart to the [translating guide](./translating/). The translating guide covers "I want to fix a string in Vietnamese" — file layout, the WidenStrings type, the `Locale` extension pattern. This page covers the **guard rails Sprint 7 added** so a future contributor (or your own future self, three months from now) doesn't accidentally break the EN/VI invariant or ship source-without-bundle drift.

## The two scripts

Sprint 7 added two new workspace scripts at the repo root. Both run from anywhere — they walk the repo by relative path.

```bash
# Mechanical gate. Fails if any leaf key present in en is missing in vi
# (or vice versa). This is the CI check. Use it before pushing.
pnpm i18n:check

# Pre-PR broom. Greps web/src for JSX text + string-literal copy that
# isn't already inside a t("...") call. Non-blocking (exits 0 always) —
# it informs, doesn't gate. Run it when adding a new feature surface.
pnpm i18n:audit
```

`pnpm i18n:check` is the canonical answer to "did I break the invariant?" — it covers the case where one locale has a leaf the other doesn't, even if the missing key is never `t()`-called yet.

`pnpm i18n:audit` is the pre-PR broom that should have caught the Sprint 3 "Settings UI source-but-not-bundle" regression at PR time, not at user-report time. It is **deliberately conservative** — false positives are tolerable (eyeball + remove), false negatives are not. The regex is tight: ≥ 3 ASCII letters inside JSX text content, OR a capitalized phrase in a JSX prop string. URLs, brand names, numeric units, and short strings are allow-listed.

## The unit test

`web/src/i18n/messages.test.ts` is the same invariant, runnable under vitest. It runs in the existing `pnpm --filter web test` step (already in the `lint-web` CI job). Three cases:

1. Both `en` and `vi` are exported and registered in the `messages` object.
2. `en` and `vi` have the same leaf-key set (the same thing `i18n:check` does, runnable under vitest).
3. No leaf value is empty in either locale.

If a future refactor moves strings to per-locale files or splits `messages.ts`, the test stays — it imports the same surface and asserts the same shape, just from a different module path.

## How to add a key (the contributor's view)

You just added a new Settings tab and need a label. Quickest path:

1. **Open `web/src/i18n/messages.ts`.** Find the `en` object → the namespace (`settings.*` here).
2. **Add the key under the right namespace.** Match the existing shape. New labels are usually 3-6 sibling keys (title / description / button + maybe a hint).
3. **Mirror the same key into `vi`.** The `vi: WidenStrings<typeof en>` type means TypeScript will refuse to compile if a key is in `en` but missing in `vi`. So a half-done translation is a compile error, not a runtime one.
4. **Run `pnpm i18n:check`** from the repo root. Should exit 0 with "686 keys present in both en and vi" (count grows with each feature).
5. **Run `pnpm i18n:audit`** from the repo root. The output is review-only — if your new feature surfaces inline English by accident, it'll show up here before the bundle-freshness job catches it on CI.
6. **Add a string literal to the `bundle-freshness` job's grep list** (`.github/workflows/ci.yml`). This is the layer that catches the "I added a new key but forgot to rebuild the bundle" regression. The next time someone refactors `messages.ts` and the key disappears from the emitted JS, CI fails instead of the user seeing a half-localized page.

The translating guide's steps 4-5 ("run `tsc --noEmit`" and "eyeball in the dev server") still apply — those catch the type system miss and the visual regression. Sprint 7 just added two more mechanical layers around the same invariant.

## How to add a third locale

The translating guide covers this in detail. The Sprint 7 additions:

- **Add the locale to `Locale`** in `web/src/i18n/types.ts`.
- **Mirror the full `en` object** as the new locale. The WidenStrings type means you can start with EN values for any untranslated key and the build still passes — but every "still-English" key is a visible gap in the UX. The audit script will flag them once the locale is the active one.
- **Register the new locale in `messages = { en, vi, ... } as const`**. The `I18nProvider` picks up new entries from this map automatically.
- **Add a `<LanguageToggle>` entry** so the operator can switch to it. `web/src/components/LanguageToggle.tsx` is one file.
- **Translate the docs site** (separate from the web app). Starlight has its own i18n config under `docs/src/content/docs/{locale}/`. The two surfaces are independent — adding a third app locale does not automatically translate the docs.

A new locale is a **maintenance commitment**, not a one-time PR. Without a translation owner, the locale goes stale within a release. The Lumen project author owns VI; any third locale needs a maintainer to sign up for ongoing translation before it ships. See [`CONTRIBUTING.md`](https://github.com/quanla93/lumen/blob/main/CONTRIBUTING.md#contributor-tiers) for the contributor tiers that govern this.

## What stays verbatim

Translation policy (per the translating guide, reinforced here for new contributors):

- **Brand names** (`Lumen`, `OIDC`, `SAML`, `S3`, `VAPID`, `SSO`) stay verbatim across all locales. The English form is the canonical one.
- **URLs, hostnames, paths, command-line flags** stay verbatim.
- **Units and unit-like strings** stay verbatim (`MB`, `GiB`, `1m`, `5s`, `…` ellipsis). Translate the *label* around them, not the unit.
- **Error codes and identifiers** (`UNAUTHENTICATED`, `cpu_pct`, `web_push`) stay verbatim. Translate any *human-readable explanation* alongside them.

Translators handle connective copy and surface the locale's natural phrasing. Operators reading the UI in their language still need to recognize the technical name — that recognition is part of the UX, not a translation miss.

## Known limitation (carried from Sprint 5/6)

The `TranslationKey` type in `web/src/i18n/types.ts` is a recursive template-literal type that walks the `en` object. TypeScript's instantiation-depth limit is finite; very deep branches (`onboarding.*`, `passkeys.*`) fall out of the union silently. Components that need those deep keys cast the hook's `t()` function once at the top of the file:

```ts
const tt = t as unknown as (k: string, p?: Record<string, string | number>) => string;
```

The cast is safe — `I18nProvider.lookup()` already handles arbitrary dotted strings at runtime. The pattern is documented in Sprint 5/6's RFCs; if a future feature lands another deep branch, copy the same cast pattern. A `LeafKeys` refactor (intersection types, `& string` constraints, or a non-recursive `K in keyof T` over the top level) is a known follow-up but out of scope for the polish pass.

## What Sprint 7 did NOT cover (follow-ups)

The audit script still flags inline English in two surfaces that were not in the RFC 0007 "5 surfaces" scope:

- **Alerts → Maintenance tab** (~13 strings: window create form, list filter, edit/delete confirm).
- **Settings → Backup** (~10 strings: target select, S3 fields, passphrase, recent list).

Both are real copy that a VI-speaking operator would hit. The pattern is identical to D2 — promote + add EN+VI under the existing `alerts.tagsTab` / `settings.apikeysTitle` namespaces, then extend `bundle-freshness` to grep for the new keys. A future 1-day sprint can copy this exact recipe. Filed as a follow-up issue; not part of the v0.7.x ship set.

**Update (Sprint 7.5, 2026-06-11)**: the two follow-up surfaces above were promoted to `alerts.maintenance.*` (27 keys) and `settings.backup.*` (27 keys) in a follow-up commit. Audit script reports zero inline English in those surfaces; the recipe from this page was applied verbatim. The two namespaces join the existing per-feature naming convention. Total key count: 686 → 740 (+54). No further admin-surface inline English remains per the audit script at this commit.
